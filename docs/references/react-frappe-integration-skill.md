# Huf React ↔ Frappe Integration Skill Guide

This document is a practical, implementation-level guide for agents who need to build **integrated software** in this codebase (React frontend + Frappe backend + realtime).

---

## 1) High-level architecture: how React is wired to Frappe

Huf runs as a Frappe app and ships a Vite/React SPA that is served from Frappe routes.

- Frontend source lives in `frontend/`.
- Build artifacts go to `huf/public/frontend` via Vite output config.
- A built HTML entry is copied into the website route file `huf/www/huf.html`.
- Frappe website route `/huf/<path:app_path>` resolves to `huf/www/huf.py` context + `huf/www/huf.html` template.

In short:

1. React is built with static asset base `/assets/huf/frontend/`.
2. Frappe serves those assets from `/assets/...`.
3. Frappe renders `huf.html` for app route `/huf/...` and injects boot/auth/CSRF globals.
4. React router uses `basename="/huf"`, so route paths align with Frappe website rules.

## 2) Yarn workflow: root vs frontend

### 2.1 Root `package.json`
The root package is an orchestrator that delegates everything to `frontend/`.

Key scripts:

- `yarn dev` → `cd frontend && yarn dev`
- `yarn install` → `cd frontend && yarn install`
- `yarn build` → `yarn build-frontend`
- `yarn build-frontend` → install frontend deps + build
- `postinstall` → install frontend deps + build

Use root yarn commands when working from repo root and you want the "single command" developer UX.

### 2.2 Frontend `package.json`
Frontend script behavior:

- `yarn dev` → runs Vite dev server.
- `yarn build` → `tsc -b` + `vite build --base=/assets/huf/frontend/` + `yarn copy-html-entry`.
- `yarn copy-html-entry` → copies generated HTML to website entry:  
  `cp ../huf/public/frontend/index.html ../huf/www/huf.html`

**Important integration detail:** this copy step is what ties the compiled SPA entrypoint to Frappe website routing.

---

## 3) Build outputs and folder roles (`www`, `public`, `assets`)

## 3.1 `huf/public/`
For Frappe apps, files here are static and served under `/assets/huf/...` after build/symlink pipeline. Vite build output target is `huf/public/frontend`.

## 3.2 `huf/www/`
Files here are website route templates/controllers.

- `huf/www/huf.py` provides `get_context` with CSRF + boot payload.
- `huf/www/huf.html` (generated by build copy step) is the SPA shell Frappe serves.
- `huf/www/agent_chat.html` is a separate standalone web page for audio chat.

### 3.3 Entry HTML naming and why it matters
The runtime entry HTML for React in website mode is **`huf/www/huf.html`**.

Even though Vite emits `index.html`, the deployment pipeline renames/copies it to `huf.html` in `www`, because Frappe resolves route `/huf` to module/file `huf/www/huf.py` + `huf/www/huf.html`.

---

## 4) Hooks and route wiring (including redirections)

`huf/hooks.py` defines route rules and renderers.

### 4.1 `website_route_rules`
Configured route mappings:

- `/huf/stream/ping` → `huf/stream/ping`
- `/huf/stream/<path:agent_name>` → `huf/stream`
- `/huf/stream` → `huf/stream`
- `/huf/docs` → `huf/docs`
- `/huf/docs/<path:app_path>` → `huf/docs/<path:app_path>`
- `/huf/<path:app_path>` → `huf`

This is effectively the redirection/routing contract:

- streaming endpoints are normalized to stream renderer path,
- docs routes are handled before catch-all,
- all other app paths fall through to SPA shell route (`huf`).

### 4.2 `page_renderer`
Custom renderer chain:

- `huf.ai.agent_stream_renderer.AgentStreamRenderer`
- `huf.www.docs_renderer.DocsRenderer`

These allow non-standard website behavior (SSE stream endpoint + docs static renderer).

### 4.3 App URL behavior
`app_url` is configured as `"huf"` (with install-time note for v16 updates). App tile route points to this URL.

## 5) React router base URL and Frappe compatibility

In `App.tsx`, React uses:

```tsx
<BrowserRouter basename="/huf">
```

This **must** match Frappe website route mounting (`/huf/...`) and generated HTML entry path behavior.

Separately, Vite build base is:

```bash
--base=/assets/huf/frontend/
```

So:

- application routes use `/huf/...`
- static JS/CSS chunks load from `/assets/huf/frontend/...`

This separation is critical and intentional.

## 6) Boot, auth, and CSRF initialization flow

## 6.1 Frappe context injection (`huf/www/huf.py`)
`get_context` does:

- generate CSRF token (`frappe.sessions.get_csrf_token()`),
- fetch boot payload (`frappe.sessions.get()` for authenticated, website boot for guest),
- sanitize script tags out of boot JSON,
- inject into template context:
  - `csrf_token`
  - `boot`
  - `build_version`

## 6.2 HTML boot globals (`frontend/index.html` template)
The HTML shell sets globals:

```html
<script>
  window.csrf_token = "{{ csrf_token }}";
  if (!window.frappe) window.frappe = {};
  window.app_name = "{{ app_name }}";
  frappe.boot = JSON.parse({{ boot }});
</script>
```

React and SDK utilities then rely on these globals.

## 6.3 Frontend SDK setup
`frontend/src/lib/frappe-sdk.ts`:

- Base URL = `VITE_FRAPPE_URL` or `window.location.origin`.
- Creates `FrappeApp` instance and exports `auth`, `db`, `call`, `file` services.

## 6.4 Auth implementation
`UserContext` performs auth check on mount:

1. `auth.getLoggedInUser()`
2. if logged in → fetches `User` doctype via `db.getDoc`
3. if not logged in → redirects to login

Redirect pattern:

```text
/login?redirect-to=%2Fhuf#login
```

Protected routes depend on `UserContext`; unauthenticated state returns null while redirect happens.

## 7) API call patterns used by React

React uses two main patterns:

1. **DocType CRUD via `db.*`** (REST resource style)
2. **Whitelisted Python methods via `call.get/post`** (`/api/method/...`)

## 7.1 Pattern A — DocType list/doc fetch
Example from chat APIs:

```ts
await db.getDocList('Agent Conversation', {
  fields: ['name', 'title', 'agent', 'modified'],
  filters: [['channel', '=', 'Chat']],
  orderBy: { field: 'modified', order: 'desc' },
  limit: 20,
});
```

Use this for standard CRUD/list queries.

## 7.2 Pattern B — whitelisted methods
Example:

```ts
await call.post('huf.ai.agent_chat.new_conversation', {
  agent: 'Support Agent',
  message: 'Hello',
});
```

This maps to `@frappe.whitelist()` function and is ideal for orchestration/business logic.

## 8) Key endpoints with response shapes and usage examples

> Note: with Frappe method APIs, payloads are usually wrapped in `message` depending on SDK method behavior.

## 8.1 Permissions / identity

### Endpoint
`GET /api/method/huf.permissions.get_me`

### Purpose
Load user identity + Huf capabilities for frontend gating.

### Typical response
```json
{
  "message": {
    "user": "jane@example.com",
    "full_name": "Jane Doe",
    "huf_role": "Huf Manager",
    "capabilities": ["agent.use", "chat.use", "flows.use"]
  }
}
```

### Example
```ts
const me = await call.get('huf.permissions.get_me');
const capabilities = me.message.capabilities;
```

## 8.2 New conversation

### Endpoint
`POST /api/method/huf.ai.agent_chat.new_conversation`

### Request
```json
{ "agent": "Support Agent", "message": "Summarize this ticket" }
```

### Response shape (from frontend typing)
```json
{
  "message": {
    "success": true,
    "conversation_id": "CONV-0001",
    "run": {
      "success": true,
      "response": "...",
      "structured": null,

      "agent_run_id": "RUN-0001",
      "conversation_id": "CONV-0001",
      "session_id": "chat:user@example.com"

```

### Example
```ts
const res = await call.post('huf.ai.agent_chat.new_conversation', {
  agent: 'Support Agent',
  message: 'Summarize this ticket',
```

## 8.5 Streaming availability check

### Endpoint
`GET /huf/stream/ping`

### Response
```json
{ "ok": true, "status": "ok" }
```

Used at app load to decide whether SSE streaming path should be enabled.

## 8.6 SSE stream endpoint

### Endpoint
`POST /huf/stream/<agent_name>`

### Request body example
```json
{
  "prompt": "Help me debug this",
  "channel_id": "Chat",
  "conversation_id": "CONV-0001"
}
```

### Event stream chunks
Each SSE frame is:

```text
data: { ...json... }
```

Chunk types used by frontend:

- `delta` (incremental/final text fields)
- `tool_call`
- `complete`
- `error`

Example `complete` chunk:

```json
{
  "type": "complete",
  "success": true,
  "response": "Final response",
  "conversation_id": "CONV-0001",
  "agent_run_id": "RUN-0003",
  "session_id": "chat:user@example.com",
  "provider": "OpenAI"
}
```

## 9) Realtime and WebSocket integration (Frappe realtime)

There are **two realtime channels** in this app:

1. **SSE** for streamed model output (`/huf/stream/...`).
2. **Socket.IO/Frappe realtime** for conversation/tool/message events.

## 9.1 Socket connection details
Frontend builds socket URL as:

```text
{protocol}://{host}:{port}/{siteName}
```

Where:

- `siteName` = `window.frappe.boot.sitename`
- `port` = `window.frappe.boot.socketio_port` when relevant
- transports prefer websocket with polling fallback

## 9.2 Conversation event channel
Frontend subscribes to:

```text
conversation:{conversationId}
```

Backend emits with `frappe.publish_realtime(event=f'conversation:{conversation_id}', ...)`.

## 9.3 Event types emitted by backend
Observed emitted types:

- `tool_call_started`
- `tool_call_completed`
- `tool_call_failed`
- `new_agent_message`
- `new_user_message`
- `update_message`

These are used by the chat UI to append messages and tool status in near-real-time.

## 10) Dev proxy and local integration behavior

`frontend/proxyOptions.ts` auto-discovers bench config and proxies Vite dev requests for:

- `/app`
- `/api`
- `/assets`
- `/files`
- `/private`
- `/login`

to detected Frappe site + webserver port.

This allows running React dev server while using Frappe backend/session/cookies seamlessly.

## 11) Docs renderer and non-SPA website handling

`DocsRenderer` serves static docs HTML from:

```text
<app>/www/huf/docs/**/index.html
```

Key behavior:

- only handles `huf/docs` paths,
- ignores assets and Next internals (`_rsc=`),
- falls back to root docs index when subpath missing.

This exists alongside SPA catch-all routing without conflict due to route ordering in hooks.

## 12) Implementation checklist for agents building integrated features

When adding a new integrated feature, follow this sequence:

1. **Choose route type**
   - SPA route under `/huf/...` or dedicated `www/*.html` page?
2. **Choose backend API style**
   - simple CRUD via `db.*` or orchestrated method via `@frappe.whitelist` + `call.*`.
3. **Wire auth + permission**
   - enforce capability checks server-side, not only UI hiding.
4. **CSRF/session correctness**
   - include credentials; include `X-Frappe-CSRF-Token` for manual fetch POSTs.
5. **Realtime need**
   - if partial generation required: SSE.
   - if event notifications required: `publish_realtime` + socket subscription.
6. **Routing updates**
   - update `hooks.py` `website_route_rules` and/or `page_renderer` as needed.
7. **Build pipeline sanity**
   - ensure generated frontend entry is copied to `huf/www/huf.html` when needed.
8. **Document endpoint contract**
   - request schema, response schema, and one frontend call example.

---

## 13) Common pitfalls

- Forgetting React basename `/huf` causes broken deep links.
- Confusing SPA base path (`/huf`) with static assets base (`/assets/huf/frontend/`).
- Editing `huf/www/huf.html` directly when it is build-generated.
- Assuming websocket is always available; app already guards with streaming/socket checks.
- Relying only on frontend permission gates; always enforce on whitelisted methods.

## 14) Quick reference (copy/paste)

### Create whitelisted method
```py
@frappe.whitelist()
def my_method(arg1: str):
    return {"ok": True, "arg1": arg1}
```

### Call from frontend
```ts
const res = await call.post('huf.some_module.my_method', { arg1: 'value' });
console.log(res.message.ok);
```

### Emit realtime event
```py
frappe.publish_realtime(
  event=f'conversation:{conversation_id}',
  message={"type": "new_agent_message", "conversation_id": conversation_id},
  user=frappe.session.user,
)
```

### Listen in React
```ts
socket.on(`conversation:${conversationId}`, (data) => {
  if (data.type === 'new_agent_message') {
    // update UI
  }
});
```
