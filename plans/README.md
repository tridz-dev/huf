## Findings considered and rejected (do NOT re-audit)

From run 1 (`540b535`):
- **`huf/ai/flow_eval.py` `safe_eval_expression` is secure by design.** AST evaluator rejects `ast.Call`/`ast.Attribute`/unsupported nodes, binds only `context`, caps length at 500. No injection path. Plan 001 adds regression tests; no vuln to fix.
- **~22 `except Exception: pass` blocks are graceful degradation** — not security-relevant in bulk. Reopen only a specific one that swallows an auth/permission error.
- **`MAX_RESPONSE_SIZE` memory pre-check gap in `http_handler.py`** — known minor gap (10MB cap, internal tool); deliberately deferred. Proper fix is streaming with incremental size accumulation.
- **DNS rebinding / TOCTOU in `validate_url`** — partially deferred, documented in plan 002's Maintenance notes (002 closes IPv6/redirect holes but does not pin the resolved IP).
- **Hardcoded-URL integration tools (`google_*`, slack, gmail)** — fixed vendor hosts, no SSRF surface. Out of scope for 002.

From run 2 (`abdd987`) — surfaces examined by an Opus security pass and judged **safe by design** (verified against live code):
- **`run_agent_sync` (`agent_integration.py:623`) / `run_agent_sync_chat` (`chat_api.py:23`) guest gate is fail-closed.** `Agent.allow_guest` defaults `0`; `_is_user_allowed` (`agent_integration.py:279-303`) enforces guest/allowlist/role checks. The residual risk is *what an opted-in guest tool can do* — that is plan 005, not the gate.
- **`http_handler.handle_get_request`/`handle_post_request` guest path is gated.** A bare guest is blocked unless a `tool_name` with `allowed_for_guest=1` is supplied (`http_handler.py:85-90`); `allowed_for_guest` defaults `0`. The outbound fetcher cannot be driven by an anonymous caller. (SSRF itself is plan 002.)
- **`get_context_for_dev` (`huf/www/huf.py:47`)** — fail-closed: throws unless `frappe.conf.developer_mode` (a non-production flag). By-design dev tool.
- **`agent_trigger.py:32` `frappe.safe_eval`** — Frappe's restricted evaluator over admin-authored trigger config (requires write access to Agent Trigger), not attacker request data. Consistent with the flow_eval verdict. Not a finding.
- **`frappe.set_user` restoration** — `flow_tool_executor.py:34-42` uses correct `try/finally`; `agent_integration.py:494-507` runs in a dedicated worker thread with its own `frappe.init/connect/destroy`, so the switch is thread-isolated. No session bleed.
- **Raw SQL via `frappe.db.sql` (sampled)** — `sdk_tools.py`, `conversation_manager.py:80`, doctype controllers all use `%s` parameter binding with tuple args. No request/user data concatenated into SQL in the sample. No injection finding.
- **`mermaid.tsx`/`chart.tsx`/`code-block.tsx` `dangerouslySetInnerHTML`** — lower-risk (Mermaid self-sanitizes, Shiki emits structural markup, chart injects CSS). Flagged for a follow-up review in plan 006's Maintenance notes; not changed. Only the raw-content `svg` sink is fixed (plan 006).
